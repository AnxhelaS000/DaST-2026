# T4.5 Comparison Against the Initial DMP

## Initial DMP vs Final DMP

The initial DMP described the project at a planning stage and focused mainly on the intended machine learning experiment, expected datasets, software tools, FAIR principles, and anticipated outputs. The final DMP became a complete documentation record of the project and required substantially more detail about data management, metadata, repositories, licences, contributors, preservation, and reuse.

## 1. Unrealistic or incomplete assumptions in the initial DMP

The initial DMP assumed that the project outputs could be described relatively simply as a machine learning workflow consisting of input data, a trained model, evaluation metrics, and visualisations. In practice, the final project produced a much larger set of research artefacts including source code, database structures, trained models, evaluation results, metadata records, documentation, and repository deposits.

The initial DMP also treated FAIR compliance at a high level and assumed that outputs would eventually receive persistent identifiers and be deposited in repositories. During the project, additional work was required to create repository deposits, assign licences, document metadata, and connect project artefacts with persistent identifiers.

Another incomplete assumption was that documentation requirements would be straightforward. The final DMP required detailed descriptions of datasets, software, repositories, access conditions, long-term preservation plans, contributors, ORCID identifiers, licences, and reuse conditions.

## 2. What the final DMP actually required

The final DMP required considerably more detailed metadata than originally anticipated. All contributors had to be documented with ORCID identifiers and project responsibilities. Data sources, generated outputs, repositories, licences, preservation periods, and access conditions had to be clearly specified.

The final DMP also required separating project outputs into distinct research objects. Instead of treating the project as a single dataset, the final documentation described the trained Random Forest model, evaluation artefacts, reused meteorological data, repository deposits, and supporting documentation separately.

In addition, the final DMP required compliance with multiple metadata and documentation standards, including RO-Crate, CodeMeta, FAIR4ML, Croissant, Model Cards, DBRepo metadata, and repository-specific metadata requirements. Repository locations, licences, persistent identifiers, and long-term preservation information also had to be documented.

## Conclusion

The main difference is that the initial DMP was primarily a planning document describing intended work, while the final DMP became a complete record of the actual project outputs and their management. The final version required concrete metadata, repository deposits, licences, persistent identifiers, contributor information, preservation plans, and reuse documentation that could not be fully specified at the beginning of the project.
