# T3.11 – Standards Overlap Analysis

## Overview

This analysis compares the five metadata standards used in the Frost Day Prediction experiment:
- **RO-Crate** – Research Object packaging and description
- **CodeMeta** – Software metadata
- **FAIR4ML** – Machine learning model metadata
- **Croissant** – Dataset metadata (JSON-LD)
- **Model Card** – Human-readable model documentation

---

## Pairwise Analysis

### 1. RO-Crate vs CodeMeta

| Category | Fields |
|---|---|
| **Shared fields** | `name`, `description`, `author` (with ORCID), `license`, `version`, `url`/`codeRepository`, `dateCreated`, `keywords` |
| **Unique to RO-Crate** | `hasPart` (links all experiment entities), `input`/`output` (data relationships), `contentSize`, `encodingFormat`, `conformsTo` (RO-Crate profile) |
| **Unique to CodeMeta** | `softwareRequirements`, `runtimePlatform`, `programmingLanguage`, `developmentStatus`, `issueTracker`, `continuousIntegration` |
| **Conflicts** | `author` vs `creator`: RO-Crate uses `author`, CodeMeta uses `creator` for the same concept. Both reference schema.org Person but use different property names, requiring manual alignment. |

**Discussion:** RO-Crate and CodeMeta share a significant overlap in basic provenance fields (name, authors, licence, version). RO-Crate focuses on packaging all experiment entities into a graph, while CodeMeta focuses specifically on software metadata. In practice, the `codemeta.json` file can be referenced as an entity within the RO-Crate, making them complementary rather than redundant.

---

### 2. RO-Crate vs FAIR4ML

| Category | Fields |
|---|---|
| **Shared fields** | `name`, `description`, `author`, `license`, `dateCreated`, `url`, input dataset reference (with DOI) |
| **Unique to RO-Crate** | `hasPart`, `conformsTo`, `encodingFormat`, `contentSize`, experiment-level packaging |
| **Unique to FAIR4ML** | `mlTask`, `mlFramework`, `hyperparameters`, `evaluationMetrics`, `inputFeatures`, `targetVariable`, `dataSplit`, `intendedUse`, `knownLimitations` |
| **Conflicts** | No direct conflicts. FAIR4ML extends the model description beyond what RO-Crate covers. RO-Crate references the FAIR4ML file as an entity, so they are hierarchically nested rather than overlapping. |

**Discussion:** RO-Crate provides the packaging layer that references the FAIR4ML file, while FAIR4ML provides the deep ML-specific metadata. Together they cover the full provenance chain from data to model. FAIR4ML fills a gap that RO-Crate cannot address with its generic schema.

---

### 3. RO-Crate vs Croissant

| Category | Fields |
|---|---|
| **Shared fields** | `name`, `description`, `license`, `url`, `author`/`creator`, dataset distribution information |
| **Unique to RO-Crate** | `hasPart` (full experiment graph), `conformsTo`, model and code entities |
| **Unique to Croissant** | `recordSet`, `field` definitions (with data types), `source` (data transformations), QUDT unit references, ML-specific field semantics |
| **Conflicts** | Both describe dataset fields but use different vocabularies. RO-Crate uses generic schema.org properties while Croissant uses ML-specific field definitions. A dataset described in both may have inconsistent field descriptions if not carefully aligned. |

**Discussion:** Croissant is specifically designed for ML dataset discovery and describes the internal structure of datasets (fields, types, units). RO-Crate describes the dataset as a whole entity within the experiment. They are complementary — Croissant provides field-level detail that RO-Crate references at a higher level.

---

### 4. RO-Crate vs Model Card

| Category | Fields |
|---|---|
| **Shared fields** | Model name, description, author, licence, training dataset reference, evaluation results |
| **Unique to RO-Crate** | Machine-readable JSON-LD graph, `hasPart` relationships, DOI references, `encodingFormat` |
| **Unique to Model Card** | Intended use, out-of-scope uses, ethical considerations, limitations (narrative), evaluation results table (human-readable) |
| **Conflicts** | No direct conflicts. Model Card is human-readable Markdown; RO-Crate is machine-readable JSON-LD. They serve different audiences and are fully complementary. |

**Discussion:** RO-Crate and Model Card target different audiences. RO-Crate enables machine-readable discovery and interoperability, while the Model Card communicates model behaviour to human readers. Both reference the same evaluation metrics but in different formats. The Model Card is referenced as an entity within the RO-Crate.

---

### 5. CodeMeta vs FAIR4ML

| Category | Fields |
|---|---|
| **Shared fields** | `name`, `description`, `author` (with ORCID), `license`, `version`, `programmingLanguage`, software dependencies |
| **Unique to CodeMeta** | `softwareRequirements`, `runtimePlatform`, `developmentStatus`, `codeRepository`, `issueTracker` |
| **Unique to FAIR4ML** | `mlTask`, `hyperparameters`, `evaluationMetrics`, `inputFeatures`, `targetVariable`, `dataSplit`, `intendedUse`, `knownLimitations` |
| **Conflicts** | No conflicts. CodeMeta describes the software package; FAIR4ML describes the ML model trained by that software. They address different aspects of the same artefact. |

**Discussion:** CodeMeta and FAIR4ML are highly complementary with minimal overlap. CodeMeta covers the software engineering perspective (dependencies, runtime, repository) while FAIR4ML covers the ML science perspective (training data, hyperparameters, metrics). Both are needed for full reproducibility.

---

### 6. CodeMeta vs Croissant

| Category | Fields |
|---|---|
| **Shared fields** | `name`, `description`, `license`, `author`/`creator`, `url` |
| **Unique to CodeMeta** | Software-specific fields: `softwareRequirements`, `programmingLanguage`, `runtimePlatform`, `codeRepository` |
| **Unique to Croissant** | Dataset-specific fields: `recordSet`, `field`, data types, QUDT unit references, distribution format |
| **Conflicts** | No conflicts. CodeMeta describes software; Croissant describes datasets. They address entirely different artefact types with minimal overlap beyond basic provenance. |

**Discussion:** CodeMeta and Croissant describe fundamentally different artefact types (software vs dataset) and therefore have very little overlap beyond basic Dublin Core-style provenance fields. In this project, CodeMeta describes the training pipeline and Croissant describes the input dataset.

---

### 7. CodeMeta vs Model Card

| Category | Fields |
|---|---|
| **Shared fields** | `name`, `description`, `author`, `license` |
| **Unique to CodeMeta** | `softwareRequirements`, `runtimePlatform`, `programmingLanguage`, `version`, `codeRepository` |
| **Unique to Model Card** | Intended use, out-of-scope uses, ethical considerations, evaluation results table, known limitations |
| **Conflicts** | No conflicts. CodeMeta is machine-readable JSON; Model Card is human-readable Markdown. They are fully complementary. |

**Discussion:** Very little overlap. CodeMeta and Model Card together provide a complete picture: CodeMeta for the software implementation and Model Card for the model behaviour and responsible AI documentation.

---

### 8. FAIR4ML vs Croissant

| Category | Fields |
|---|---|
| **Shared fields** | Dataset reference (training data), field names and data types, licence of training data |
| **Unique to FAIR4ML** | `mlTask`, `hyperparameters`, `evaluationMetrics`, `dataSplit`, `intendedUse`, `knownLimitations`, model-level metadata |
| **Unique to Croissant** | `recordSet`, field-level data type definitions, QUDT unit URIs, dataset distribution information, data transformations |
| **Conflicts** | Both describe dataset fields but at different levels. FAIR4ML lists input features by name; Croissant provides full field definitions including data types and units. If field names or descriptions differ between the two, this creates an inconsistency that must be manually reconciled. |

**Discussion:** FAIR4ML and Croissant have a meaningful overlap in their description of the training dataset's features. Croissant provides the authoritative field-level description while FAIR4ML references those fields as model inputs. Consistency between the two must be maintained carefully — the same field names and units should appear in both.

---

### 9. FAIR4ML vs Model Card

| Category | Fields |
|---|---|
| **Shared fields** | Model name, algorithm, training dataset (with DOI), evaluation metrics (accuracy, precision, recall, F1, ROC-AUC), intended use, known limitations, licence |
| **Unique to FAIR4ML** | `mlFramework`, `hyperparameters` (machine-readable), `dataSplit` details, `inputFeatures` (structured list) |
| **Unique to Model Card** | Out-of-scope uses, ethical considerations, evaluation results table (Markdown), narrative descriptions |
| **Conflicts** | Both contain evaluation metrics and intended use descriptions. If values or descriptions diverge between the machine-readable FAIR4ML and the human-readable Model Card, this creates an inconsistency. Metrics must be identical in both. |

**Discussion:** FAIR4ML and Model Card have the most significant overlap of all pairs. They both document the same model from different perspectives: FAIR4ML provides structured, machine-readable metadata while the Model Card provides human-readable documentation. Both are needed — FAIR4ML for automated discovery and interoperability, Model Card for transparency and responsible AI communication. Evaluation metric values must be kept in sync between the two.

---

### 10. Croissant vs Model Card

| Category | Fields |
|---|---|
| **Shared fields** | Training dataset name, description, licence |
| **Unique to Croissant** | `recordSet`, field definitions, data types, QUDT unit URIs, distribution format, data transformations |
| **Unique to Model Card** | Training data section (narrative), evaluation results, intended use, ethical considerations, limitations |

**Discussion:** Minimal overlap. Croissant provides the technical dataset specification while the Model Card references the dataset narratively in the training data section. They are complementary with no conflicts.

## Key Findings

1. **FAIR4ML and Model Card** have the most overlap — both document evaluation metrics and intended use. They must be kept in sync to avoid inconsistencies.

2. **RO-Crate acts as the packaging layer** that references all other standards as entities. It has overlap with every standard but serves a different purpose — it connects them all into a single machine-readable graph.

3. **CodeMeta and Croissant** have the least overlap — they describe fundamentally different artefact types (software vs dataset).

4. **The main conflict** across all standards is the `author` vs `creator` property naming between RO-Crate and CodeMeta. Both reference schema.org Person but use different property names, requiring manual alignment.

5. **FAIR4ML and Croissant** share dataset field descriptions — field names and units must be consistent between the two to avoid interoperability issues.
